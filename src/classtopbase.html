Dashboard Design

As our team add more features to ClassTopBase, dashboard is the natural thing we started to think about developing next. I have to admit that we don’t know how useful dashboard will be the users, but since it’s relatively easy to develop, we figured to do it first and see how it works. The process for developing dashboard as the following. Brain storming, card sorting, wireframe, design, build, test, go live.

Brain Storming

Me and my product team sit together. We listed everything ClassTopBase does on sticky notes, and separate them into three categories. Features, Navigations and Metrics.

Card Sorting

Then we draw a big triangle on a white board as the basic shape of dashboard. We then place all the sticky notes onto the triangle based on where we think they belong. During this process, I will challenge the team, including me, the placement of each sticky note. Whether they are useful to users or not. There were a lot of back and forth, debating what should be on dashboard and where should it be. From doing this, we will get a general wireframe for the dashboard.

Wireframing

The brain storming and card sorting exercise were helpful. However, the outcome is still too complex. So I took my time and nail down five things to show on the dashboard.

  1. 1. Numeric status on how many students, teachers this school has.
    Giving a quick and general idea about the school.
  2. 2. All the check-in status about students.
    Show how school is running and whether students attend school.
  3. 3. Status on how many students registered from different channels.
    Indicating how well school at recruiting new students.
  4. 4. Today’s class schedule.
    Give a glaze on what classes will happen today, so staff can take action right on the dashboard.
  5. 5. Recent activity.
    Staff will know what happened in recent management activities.

Design & Building

The design for the dashboard is relative simple. I tried different usage of the colors. I chose the one with more colors because they are easy to distinguish and visually more interesting.

New Navigation and UI Overhaul

New Navigation

The old navigation is not particular easy to use. The main categories and the sub categories are separate from each other. It requires users to move mouse quite far. With more features we have, we need to do a redesign of the navigation. We were also not happy with old UI design. Because the developer team rushed to build the product, UI are not consistent through out the interface. So I was tasked to redesign the navigation and UI.

I chose a popular design pattern for the navigation. Put every link to the left and let user click through it. People have seen this kind of navigation before, so it’s easy for them to pick up and it can include all of our features. It is also future-proof, it can fit more links to it.

I was choosing between a darker background for the navigation or a lighter one. I eventually decided on a darker color, because it has better separation between the navigation and main content.

I first let user click through the navigation. Once developer finished building the navigation, I was messing around with CSS and try to find a better way to use the navigation. I tried to let user hover on the navigation to expend the sub-categories. Although I like hover on things myself, it is not user friendly. Hover state can be easily triggered. The navigation will jump around because of that. So eventually I went back on the old click through method.

UI Overhaul

When I was doing the UI overhaul, I chose the blue color we had in the old design as the brand color. Blue used a lot in SaaS products and banks, because people think blue means safe and security. We want to convey the same idea. I adjust that blue to a dark grade to be AA compliant defined by WCAG 2.0 guidelines for contrast accessibility. 

When I started working on designing the interface, I went straight to CSS and HTML. It’s faster for me to try different things and see it in the browser. However, later after I finished everything, I realized that’s was not the best idea. I saved sometimes on design the UI. However, there is no design file that contain the new design. If I want to collaborate with other designers, I will have problems. It’s fine for now, we build most things in the product with component, so when we make new features, I can just let the developers know to use existing components.
